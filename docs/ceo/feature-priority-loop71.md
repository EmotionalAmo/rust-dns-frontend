## Working Backwards: 用户场景还原

设好 rust-dns，配了规则，等了一天。然后呢？

**用户的核心焦虑只有一个："它在工作吗？"**

把三个功能放到这个焦虑面前：

### Feature B: DNS 查询日志

用户打开日志页面，看到一条条实时查询记录：`doubleclick.net -> BLOCKED`，`github.com -> ALLOWED`。焦虑立刻消除。不仅知道"在工作"，还知道"怎么工作的"。遇到网页打不开，搜一下域名，秒级定位是不是被误杀。这是 Debug 的终极工具，也是信任建立的关键界面。

**场景：** 用户刚配好 rust-dns，打开日志页，看到广告域名被拦截 -- "好，它在干活。" 某个网站加载异常，搜索域名，发现被规则误杀，点击放行。闭环。

### Feature A: 规则命中统计

用户看到某条规则命中 1,247 次，另一条 0 次。知道哪些规则有用，哪些是废物。但这只回答了"哪条规则在工作"，不回答"刚才那个请求怎么了"。无法 Debug，无法处理误杀。是日志的聚合视图，但缺少原始数据时聚合没有根基。

**场景：** 用户看到命中数，点点头 -- "嗯，在工作。" 但网页打不开时，命中统计帮不上忙。

### Feature C: Dashboard 完善

图表好看，但信息本质上是 A 和 B 的可视化。没有底层数据（日志），图表画什么？先有数据，再有图表。顺序不能反。

**场景：** 用户看到一条漂亮的曲线。然后呢？"那又怎样？这告诉我什么？" 如果没有钻取到具体查询的能力，图表就是装饰。

## "What won't change?"

DNS 过滤用户永远需要：
1. **可见性** -- 我想看到发生了什么（日志）
2. **控制权** -- 我想改变规则（已有 Inline Quick-Add）
3. **简单** -- 一眼看懂，不用学

日志是可见性的根基。没有日志的 DNS 过滤器是黑箱。

---

## Priority Decision

**先做 B: DNS 查询日志。**

三个理由：

1. **解决核心焦虑的唯一方式。** 命中统计告诉你"规则在工作"，日志告诉你"每一个请求发生了什么"。后者包含前者的信息量，反过来不行。

2. **闭环 Dogfooding。** 现在 Dogfooding 阶段遇到问题，只能看后端日志。前端有查询日志后，Debug 路径从"SSH 到服务器 -> grep 日志"变成"打开浏览器 -> 搜索域名"。这直接加速飞轮：更好的 Debug 体验 -> 更快发现问题 -> 更快改进规则 -> 更好的过滤效果。

3. **差异化机会。** AdGuard Home 的日志页面臃肿且慢。rust-dns 内存只有 4.6MB，日志页面可以做到极致轻量和快速响应。这是我们的结构性优势能直接转化为用户体验优势的地方。

命中统计（A）是日志的自然衍生 -- 有了查询记录数据，命中计数只是一个 COUNT GROUP BY。所以 B 做完后 A 几乎免费。C 则等数据层成熟后再做，否则是空中楼阁。

**这是双向门决策，风险可控。**

## PR/FAQ: DNS 查询日志

### Press Release (内部)

**rust-dns 新增实时查询日志 -- 每一条 DNS 请求，一目了然**

rust-dns v1.7.0 发布查询日志功能。用户可以在前端实时查看所有 DNS 查询记录，包括域名、查询类型、来源客户端、处理结果（Allowed/Blocked/Cached）和响应时间。支持按域名搜索、按状态过滤。整个页面保持 rust-dns 一贯的轻量风格 -- 加载快、占用少、信息密度高。

### FAQ

**Q1: 日志会不会让内存占用暴增，丧失轻量优势？**

不会。日志存储在 SQLite（后端已有），前端分页加载，默认展示最近 100 条。设置可配置的保留天数（默认 7 天），自动清理。内存影响可控在 1-2MB 以内。轻量是我们的护城河，任何功能都不能破坏它。

**Q2: 和 AdGuard Home 的日志功能有什么区别？**

更快、更轻、更聚焦。AdGuard Home 的日志页面加载慢且信息过载。我们做三件事做到极致：搜索一个域名、看它被放行还是拦截、一键加规则。不做花哨的统计图表堆叠 -- 那是 Dashboard 的事。

**Q3: 已经有 QueryLogs.tsx 页面了？**

前端已有 QueryLogs.tsx 文件骨架。本轮任务是实现完整功能：对接后端日志 API、实现搜索/过滤/分页、展示查询详情。如果后端 API 尚未就绪，需要同步推进。
