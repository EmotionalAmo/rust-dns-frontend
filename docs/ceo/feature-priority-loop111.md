# Feature Priority Decision: Loop 111

**Date:** 2026-03-08
**Decision by:** CEO (Bezos Framework)
**Status:** DECIDED

---

## Context

rust-dns 已经走过了"能用"阶段，进入"好用"阶段。v1.23 实现了设备活动时间线，加上之前的 Daily Digest、Domain Check、Cache 统计、Upstream Failover 日志，核心的**可见性基础设施已经相当完整**。

当前用户：1 人，14 台设备。产品处于 Dogfooding 深水区 -- 每天在用，痛点已经不是"看不到发生了什么"，而是"维护太麻烦"和"能不能更智能"。

这决定了本轮候选方向的评估逻辑：**从可见性转向自动化和智能化**。

## Working Backwards: 用户的日常是什么样的？

一个每天使用 rust-dns 的人（就是 Amo），他的典型行为模式：

**早上：** 打开 Dashboard 扫一眼，确认一切正常。偶尔看看 Daily Digest。

**遇到问题时：** 打开 Query Logs 搜域名，判断是否误杀。Domain Check 验证规则匹配。

**维护时：** 手动去更新 filter lists。手动检查哪些设备行为异常。手动管理规则。

关键洞察：**"遇到问题时"的工具已经很好了。"维护时"的工具几乎没有。**

维护是摩擦。摩擦是飞轮的敌人。

## 逐一评估

### 1. Filter List 自动更新

**客户问题：** "我每隔几天就要手动去更新 filter lists，万一忘了，新的广告域名就漏过去了。"

这是一个**确定性问题**。不是"可能需要"，是"每次都要做"。Pi-hole 和 AdGuard Home 都有这个功能 -- 这不是差异化，这是**基线期望**。没有自动更新的 filter list 管理，就像一个闹钟 app 需要你每天手动设置一样。

**飞轮贡献：** 减少维护摩擦 -> 规则始终最新 -> 拦截效果持续好 -> 更信任系统 -> 更少干预。这是飞轮的**润滑剂**。

**开发成本 vs 价值：** 低成本，高确定性价值。后端加一个 cron job 逻辑 + 前端加一个设置项。没有技术风险。

**后悔测试：** 每次手动更新 filter list 时都会想"为什么这不是自动的"。是的，会后悔不早做。

### 2. Query Log 高级搜索

**客户问题：** "我想看某台设备在凌晨 2 点到 4 点之间访问了哪些被拦截的域名，现在做不到。"

有价值，但使用频率低。大部分时候简单搜索就够了。高级搜索是"偶尔需要深入调查时的工具"，不是日常工具。

**飞轮贡献：** 改善 Debug 体验，但只在特定场景下触发。

**开发成本 vs 价值：** 中等成本，中等价值。需要前端组合筛选器 UI + 后端查询优化。不紧急。

### 3. DNS over HTTPS/TLS

**客户问题：** "我的 DNS 查询在到达上游之前是明文的，ISP 可以看到。"

技术上重要，但对于家庭网络场景，优先级没那么高。当前的上游已经是可信的（Cloudflare/Google），内网到 rust-dns 本身就是局域网。DoH/DoT 是"正确的事情"，但不是"现在最痛的事情"。

**飞轮贡献：** 几乎为零。不影响日常使用体验。

**开发成本 vs 价值：** 高成本（TLS 握手、证书管理、连接池），低即时价值。这是一个**单向门决策** -- 做了就要永远维护 TLS 栈。暂缓。

### 4. 周报 / 月报

**客户问题：** "我想知道这周和上周比，拦截率是涨了还是跌了？有没有新的异常？"

Daily Digest 已经在做每日摘要了。周报是 Daily Digest 的自然延伸，但价值增量不大。除非用户群扩大到需要给非技术用户看报告，否则 Dashboard + Daily Digest 已经覆盖了这个需求。

**飞轮贡献：** 边际。

### 5. 批量规则导入/导出

**客户问题：** "我想从 Pi-hole 迁移过来，但有几百条自定义规则要重新输入。"

对于**获取新用户**来说，这是关键的**迁移摩擦消除器**。但当前用户只有 1 人，且已经在用了。如果目标是扩大用户群，这个功能是必要的。如果目标是做好 Dogfooding，这个可以等。

**时机判断：** 现在不是最优时机。等产品更成熟、准备推广时再做。

### 6. 设备风险评分

**客户问题：** "14 台设备，我想知道哪台在访问可疑域名，但我不想一台一台去翻日志。"

这很有想象力，但有几个问题：
- "风险"怎么定义？需要恶意域名数据库，需要行为基线，需要误报控制
- 开发成本高，且效果取决于算法质量
- 过早做容易做成"看起来智能但实际不准确"的东西

**飞轮贡献：** 如果做好了，非常高 -- 从被动监控变主动预警。但做不好就是噪音。

**判断：** 这是一个值得投资的方向，但需要分阶段。可以先做一个**简化版** -- 不叫"风险评分"，而是"异常检测"：比如某设备突然查询量暴增、或者首次访问大量新域名。简单的统计异常，不需要 ML。

### 7. Query Log 实时流

**客户问题：** "我想看 DNS 查询实时滚动，像 tail -f 一样。"

这是一个很"酷"的功能，Debug 时有用。但坦率说，刷新 Query Logs 页面就能看到最新记录。WebSocket 实时流的开发成本（WebSocket 基础设施、前端流式渲染、连接管理）相对于"少按一次刷新按钮"的价值来说，不划算。

**飞轮贡献：** 低。酷 != 有用。

---

## "What won't change?"

DNS 管理用户永远需要：
1. **拦截要准** -- filter lists 必须是最新的（-> 自动更新）
2. **维护要少** -- 系统自己照顾自己（-> 自动化）
3. **异常要知道** -- 出问题了要主动告诉我（-> 智能检测）

这三件事 10 年后也不会变。

## Priority Decision

### Top 3

| Priority | Feature | Rationale |
|----------|---------|-----------|
| **P0** | **Filter List 自动更新** | 消除最频繁的维护摩擦，是基线功能的补全，低成本高确定性 |
| **P1** | **Query Log 高级搜索** | 深度调查的必备工具，提升 Query Logs 页面从"能用"到"好用"的关键一步 |
| **P2** | **设备异常检测（简化版风险评分）** | 从被动监控迈向主动预警的第一步，14 台设备场景下刚好有意义 |

### 被暂缓的及理由

| Feature | 理由 | 重新评估时机 |
|---------|------|-------------|
| DNS over HTTPS/TLS | 高成本单向门，当前场景不紧迫 | 当准备面向外部用户发布时 |
| 周报 / 月报 | Daily Digest 已覆盖，边际价值低 | 用户群扩大后 |
| 批量规则导入/导出 | 当前无迁移需求，等推广阶段 | 准备 public launch 时 |
| Query Log 实时流 | 酷但不解决实际痛点，刷新即可 | 有明确用户反馈要求时 |

## PR 新闻稿（假设已发布）

### P0: Filter List 自动更新

> **rust-dns 支持 Filter List 自动更新 -- 设定一次，永不过期。** 用户可以为每个订阅的 filter list 设置自动更新频率（每天/每周/自定义），rust-dns 会在后台静默拉取最新规则，确保广告和恶意域名拦截始终与上游同步。零维护，零操心。

### P1: Query Log 高级搜索

> **rust-dns Query Logs 新增高级搜索 -- 14 台设备、百万条记录，精准定位任何一条查询。** 支持按客户端、域名模式、时间范围、查询状态（Blocked/Allowed/Cached）的任意组合过滤，并可保存常用搜索条件。从"翻日志"变成"问日志"。

### P2: 设备异常检测

> **rust-dns 新增设备异常检测 -- 你的网络守夜人。** 当某台设备的 DNS 行为出现统计异常（查询量突增、首次访问大量未知域名、拦截率异常升高），rust-dns 会自动标记并在 Dashboard 高亮展示。不需要你去找问题，问题来找你。

## 关键风险识别

| 风险 | 类型 | 应对 |
|------|------|------|
| Filter List 自动更新拉取失败（网络中断） | 双向门 | 保留上一版本，Dashboard 显示更新状态/失败告警 |
| 高级搜索性能问题（大量日志时查询慢） | 双向门 | SQLite 索引优化，前端分页，设置查询时间范围上限 |
| 异常检测误报过多 | 双向门 | 先保守阈值，只标记不告警，积累数据后迭代 |

所有三个都是**双向门决策** -- 做了不满意可以调整，不需要完美方案就可以开始。
