# rust-dns v1.4.0 功能优先级分析

**日期**: 2026-03-03
**版本**: v1.4.0
**产品阶段**: 成长期
**目标用户**: 自托管爱好者、小型技术团队、家庭用户

   - 使用频率：中高

## 候选功能评估

### 1. Upstream 配置热重载

**功能描述**: 无需重启服务即可更新 upstream 服务器配置

#### 用户价值分析

| 维度 | 评分 | 分析 |
|------|------|------|
| 可发现性 | 5/5 | 用户在添加/编辑 upstream 时会自然期望立即生效 |
| 可供性 | 5/5 | 操作后自动重载，无需额外操作步骤 |
| 心智模型匹配 | 5/5 | 符合现代 SaaS/云服务"即时生效"的期望 |
| 错误预防 | 4/5 | 配置错误时可以回滚，不会中断服务 |
| 真实需求 | 5/5 | 用户频繁测试不同 upstream，重启是严重干扰 |

#### 用户场景

1. **测试不同上游**: 用户想尝试 Cloudflare vs Google DNS，需要快速切换
2. **故障切换**: 当发现某个 upstream 延迟高时，立即切换到备用
3. **动态调整**: 根据网络状况调整 upstream，不想中断所有 DNS 请求

#### 潜在问题

- 热重载时如何处理正在进行的 DNS 查询？
- 配置错误时如何回滚？
- 用户是否需要"预览模式"先验证配置？

**结论**: 高优先级。这是对现有配置管理流程的重大改进，直接影响用户日常操作体验。

---

### 2. DoT/TCP Connection Pooling

**功能描述**: 高并发场景下的连接复用优化

#### 用户价值分析

| 维度 | 评分 | 分析 |
|------|------|------|
| 可发现性 | 1/5 | 后端优化，用户感知不到 |
| 心智模型匹配 | N/A | 不可见特性 |
| 错误预防 | N/A | 不涉及用户操作 |
| 真实需求 | 2/5 | 自托管场景通常并发量有限 |
| 性能提升 | 3/5 | 对家庭用户收益小，对小型团队有收益 |

#### 用户场景

1. **小型办公场景**: 20-50 台设备同时在线，可能看到改善
2. **高流量家庭**: 多人同时流媒体、游戏
3. **监控瓶颈**: 适用于作为更大基础设施的组件

#### 市场现实

- rust-dns 的目标用户（自托管者）通常不会有高并发需求
- 即使家庭全屋设备（50+），DNS 查询频率也远低于商业场景
- 这是"技术债偿还"而非用户驱动的需求

**结论**: 低优先级。技术正确但对当前用户价值有限。可作为性能优化里程碑的"填充项"。

---

### 3. Query log 按 Upstream 筛选

**功能描述**: 在 Query Logs 页面增加 upstream 维度的筛选能力

#### 用户价值分析

| 维度 | 评分 | 分析 |
|------|------|------|
| 可发现性 | 4/5 | 现有筛选器旁边新增，用户容易发现 |
| 可供性 | 5/5 | 下拉选择，操作直观 |
| 心智模型匹配 | 5/5 | 用户已经习惯按 domain/client/status 筛选，upstream 是自然扩展 |
| 错误预防 | 5/5 | 筛选器操作不可破坏性 |
| 真实需求 | 4/5 | 高级用户诊断 upstream 问题时需要 |

#### 用户场景

1. **验证 upstream**: 添加新 upstream 后，查看哪些查询使用了它
2. **故障诊断**: 某个 upstream 有问题时，快速定位受影响的域名
3. **性能分析**: 比较不同 upstream 的响应时间和成功率
4. **流量分布**: 了解哪些 upstream 承担了主要流量

#### 设计考虑

- 需要获取 upstream 列表（已有 API）
- 筛选器应该显示 upstream 名称而非 ID
- 考虑"无 upstream"（缓存/本地处理）的特殊情况
- 实时日志面板也需要同步支持

**结论**: 中高优先级。完善现有功能，提升调试能力，但不是"关键路径"。

## 功能优先级排序

```
## 功能优先级排序

1. Upstream 配置热重载 — 核心用户体验提升
   直接影响用户最频繁的操作场景（添加/测试 upstream）
   符合"即时生效"的心智模型
   实现成本可控（后端信号机制 + 前端状态反馈）
   关键价值：消除重启带来的服务中断，降低使用门槛

2. Query log 按 Upstream 筛选 — 完善监控能力
   自然扩展现有筛选器功能，学习成本低
   支持高级用户的调试和性能分析需求
   前端工作量小（API 已支持，只需 UI）
   关键价值：提升可观察性，帮助用户做更好的 upstream 选择

3. DoT/TCP Connection Pooling — 性能优化（延后）
   用户感知度低，对当前目标用户价值有限
   属于"技术正确"但"商业价值不明"的功能
   可作为性能优化 Sprint 的内容，而非功能版本
   关键价值：为未来扩展（更大型部署）打基础，但不是当前重点
```

## 产品建议

### 前端需要配合的功能

#### 对于 Upstream 配置热重载

**需求状态**: 必须实现

1. **重载进度指示**
   - 保存 upstream 配置时显示"配置已保存，正在应用..."
   - WebSocket 或轮询获取重载状态
   - 失败时提供详细错误信息和重试按钮

2. **不可编辑期间的 UI 状态**
   - 热重载期间禁用 upstream 相关的所有编辑操作
   - 使用 loading 骨架屏防止用户困惑

3. **操作反馈优化**
   ```
   成功场景:
   - Toast: "配置已保存并应用成功"
   - Upstream 列表自动刷新（已有 30s 轮询，可优化为即时）

   失败场景:
   - Toast: "配置保存成功但应用失败: [具体错误]"
   - 提供"重试应用"按钮
   - 配置保持在待应用状态，标记为"pending"
   ```

4. **配置回滚机制**（可选但推荐）
   - 保存最近 5 个配置版本
   - 失败时提供"回滚到上一版本"选项
   - 类似 Git 的版本历史 UI

#### 对于 Query log 按 Upstream 筛选

**需求状态**: 必须实现

1. **API 扩展**
   ```typescript
   // src/api/queryLog.ts
   export interface QueryLogListParams {
     limit?: number;
     offset?: number;
     domain?: string;
     status?: 'blocked' | 'allowed';
     client?: string;
     upstream?: string;  // 新增
   }
   ```

2. **UI 组件调整**
   - 在现有筛选器区域增加 upstream 下拉框
   - 下拉选项来自 upstreamsApi.list()
   - 包含"全部"（默认）和"无 upstream"（缓存/本地）选项
   - 实时日志面板同步显示 upstream 列

3. **显示优化**
   - upstream 列已有数据，无需额外处理
   - 考虑为长的 upstream 名称添加 tooltip
   - 统计面板可以考虑按 upstream 分组的饼图（可选）

4. **i18n 扩展**
   ```json
   // locales/zh-CN.json
   {
     "queryLogs.filterUpstream": "上游服务器",
     "queryLogs.upstreamAll": "全部",
     "queryLogs.upstreamNone": "本地/缓存",
     "queryLogs.upstreamPlaceholder": "选择 upstream"
   }
   ```

#### 对于 DoT/TCP Connection Pooling

**需求状态**: 无需前端配合

- 纯后端优化
- 用户无感知
- 如需展示，可在设置页面增加"连接池状态"面板（低优先级）

## 发布策略建议

### v1.4.0 最小可行版本

**包含功能**:
- Upstream 配置热重载
- Query log 按 Upstream 筛选

**理由**:
- 两个功能都直接面向用户，有明确的使用场景
- 前端工作量适中（~2-3 天）
- 可以打包为一次有意义的版本发布

### v1.4.1 性能优化版本

**包含功能**:
- DoT/TCP Connection Pooling
- 其他性能指标优化

**理由**:
- 与功能性更新分开发布
- 可通过 changelog 向技术用户展示性能提升
- 适合在用户反馈性能问题后发布

## 风险与缓解措施

### Upstream 配置热重载风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 重载失败导致配置丢失 | 高 | 保持配置文件，失败时回滚到上一版本 |
| 部分应用（部分 upstream 更新） | 中 | 采用原子更新或全量重载 |
| 并发编辑冲突 | 低 | 前端锁定机制，同时只允许一人编辑 |

### Query log 筛选风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 大量 upstream 导致下拉框过长 | 低 | 搜索框或分页 |
| upstream 名称变更导致历史数据失效 | 低 | 使用 ID 而非名称存储 |
