# 功能优先级分析 — Loop 71
**角色：** interaction-cooper
**日期：** 2026-03-06
**阶段：** Dogfooding（开发者自用）

---

## Primary Persona 确认

**Alex — 自托管极客**

| 维度 | 描述 |
|------|------|
| Life Goal | 掌控自己家庭网络，不依赖第三方服务 |
| Experience Goal | 一眼看出系统状态，出问题能快速定位 |
| End Goal | 广告被拦截，同时能 Debug 异常流量 |

Alex 的核心焦虑是：**"我不知道它是否真的在工作。"**
其次是：**"出了问题我怎么查？"**

---

## 三个方向的 Alex 视角分析

### 方向 B：DNS 查询日志（1st）

**Alex 的使用场景：**

> 晚上 Alex 发现某个设备的流量有点奇怪，或者某个网站突然打不开。他打开 rust-dns，想知道到底发生了什么。Dashboard 的聚合数据给不了答案——他需要看原始记录：谁在什么时间请求了什么域名，结果是 Block 还是 Pass。

这是 Alex 真正的 End Goal 的直接支撑。DNS 过滤工具的本质是一个"决策代理"，查询日志就是这个代理的决策流水账。**没有日志，系统就是一个黑盒。** Alex 自托管的核心驱动力之一，就是为了摆脱黑盒。

**为什么排第一：**
- 唯一能回答"谁在访问什么"这个核心问题的功能
- Debug 场景（网站打不开、设备异常）下的刚需入口
- 规则管理的决策依据——看到日志才知道该加什么规则
- 没有日志，Dashboard 的聚合数据意义减半（你知道拦截了 42%，但你不知道拦截了什么）

**关键交互设计挑战：**

1. **信息密度 vs 可读性**：每条日志有 5+ 个字段，列表会很拥挤。核心挑战是默认展示什么、折叠什么——时间、域名、客户端、Block/Pass 状态必须在一行内清晰可见，延迟可以次要显示。

2. **实时 vs 历史的切换**：Alex Debug 时需要实时流，但查历史时需要静态列表。这是两种不同的心智模型，不能用同一个交互范式强行合并。建议：默认静态列表（可手动刷新），提供一个明确的"实时模式"开关。

3. **过滤器的优先级**：Alex 最常用的过滤维度是"按域名搜索"和"只看 Block"，这两个应该是 Primary UI，其他过滤条件（时间范围、客户端 IP）可以折叠为 Advanced Filter。不要一上来就展示 6 个下拉框。

4. **从日志到操作的跳转**：当 Alex 看到一条被错误拦截的记录，他的下一步是"加白名单"。这个操作应该直接在日志行内完成，不应该要求他去 Rules 页面手动输入域名。

---

### 方向 A：规则命中统计（2nd）

**Alex 的使用场景：**

> Alex 订阅了三个规则列表，加了十几条自定义规则。几个月后他回来清理，想知道哪些规则是"死规则"——从没命中过，或者哪条规则撑起了 80% 的拦截量。

这个功能解决的是规则维护的痛点，是一个"归因"工具。Alex 是 Power User，会关心这个。但它是在他已经有了查询日志的基础上，做的进一步精细化管理。

**为什么排第二：**
- 服务于规则维护场景，但这个场景的频率远低于 Debug 场景
- 没有查询日志的情况下，命中统计是孤立信息——你知道一条规则命中了 42 次，但不知道具体是什么请求触发的
- 对 Dogfooding 阶段的 Alex 来说，先搞清楚"系统在做什么"比"优化规则"更紧迫

**关键交互设计挑战：**

1. **数字的上下文**：单独显示"42 次命中"意义模糊。需要提供参照系：占总拦截量的百分比、最近一次命中时间。数字必须有上下文才能驱动决策。

2. **零命中规则的处理**：这类规则是"可能该删掉"的候选。但不能直接推断——也可能是规则在保护 Alex 从未访问过的威胁类别。UI 可以标记零命中规则，但措辞要中性，不能是"该删除这条规则"。

3. **规则列表的排序**：命中统计出来之前，规则列表默认按什么排？加了统计之后，排序维度变多了（按命中量、按添加时间、按规则来源），需要一个清晰的默认排序策略，否则用户会迷失。

---

### 方向 C：Dashboard 完善（3rd）

**Alex 的使用场景：**

> Alex 偶尔打开 Dashboard 扫一眼，确认系统还在正常跑。

Dashboard 是一个"状态确认"工具，不是一个"行动入口"。Alex 看 Dashboard 的时候，他的目标是"一切正常就关掉"，而不是"发现问题然后解决"。

**为什么排第三：**
- 当前 Dashboard 已经能回答"系统是否在工作"这个核心问题（总查询量 + 拦截率）
- 时间线图表、历史趋势是"锦上添花"——对 Alex 的 End Goal 贡献边际递减
- 按客户端统计有价值，但这个需求可以通过查询日志的过滤功能来满足，不需要单独在 Dashboard 做
- Dogfooding 阶段，Alex 更需要能 Debug 的工具，而不是更漂亮的图表

**关键交互设计挑战（如果要做）：**

1. **图表的信息层级**：时间线图表很容易变成"看起来很专业但没人看"的装饰品。如果要加，必须明确它能帮 Alex 发现什么——比如"流量异常尖峰"，并且要让这个发现变得显眼，而不是让 Alex 自己盯着图表找规律。

2. **历史趋势的时间窗口**：Alex 关心的颗粒度是"今天 vs 昨天"还是"这个月 vs 上个月"？不同时间窗口对应完全不同的使用场景，不能用一个默认值搞定所有人。

---

## 最终排序与结论

| 排名 | 方向 | 理由 |
|------|------|------|
| 1st | B — DNS 查询日志 | 直接支撑 Alex 的 End Goal，是其他功能的决策基础，Dogfooding 阶段的刚需 |
| 2nd | A — 规则命中统计 | 有价值但依赖日志的存在，解决的是次级场景（规则维护） |
| 3rd | C — Dashboard 完善 | 现有 Dashboard 已满足基本需求，继续堆砌是边际递减 |

**核心结论：先做 B，再做 A，C 缓行。**

DNS 查询日志是整个产品的"神经系统"——没有它，其他功能都是在盲飞。Alex 在 Dogfooding 阶段最需要的不是更好看的 Dashboard，而是能让他**理解系统行为**的原始数据入口。

---

## 方向 B 的交互设计优先级建议（给下一步参考）

如果确认做 B，以下是关键交互决策的优先级排序：

1. 日志列表的默认列（域名 + Block/Pass 状态 + 时间 + 客户端 IP）
2. 从日志行直接触发"加入白名单/黑名单"的 Quick Action
3. 域名关键词搜索 + Block/Pass 状态筛选（两个 Primary Filter）
4. 分页 vs 虚拟滚动的选择（日志量可能很大，需要明确策略）
5. 实时刷新模式（可以是 v2 功能，先做好静态列表）
